# 混合索引引擎（Hash + RBT + LinkedList）

**Meta-Index** 是一个专为 **分布式文件系统（类 HDFS）** 设计的元数据索引子系统。为了解决海量文件路径查找中的哈希冲突和性能退化问题，本项目创新性地设计并手写了 **三级混合索引架构**：**哈希表（O(1)）→ 红黑树（O(log N)）→ 链表（O(K)）**。该设计确保了在极端冲突场景下，系统性能依然稳定可控。

------

##  核心设计动机：为什么要“叠三层”？

在分布式文件系统中，元数据（文件名 → Inode）的查询 QPS 直接影响整个集群的吞吐量。传统的 `HashMap` 在面对 **海量 Key 或恶意 Hash 攻击** 时，冲突链表会无限变长，导致最坏情况查询退化为 O(N)。为了彻底解决这个问题，我设计了三级防御架构：

| 索引层级   | 数据结构                  | 时间复杂度   | 职责与触发条件                                               |
| :--------- | :------------------------ | :----------- | :----------------------------------------------------------- |
| **第一级** | **哈希表 (HashTable)**    | **O(1)**     | **快速路由**：对文件名/路径进行哈希计算，直接定位到对应的桶（Bucket）。正常情况下，这里是查询的终点。 |
| **第二级** | **红黑树 (RedBlackTree)** | **O(log K)** | **冲突分治**：当多个不同的 Key 落入同一个哈希桶（发生冲突）时，**不使用传统的链表**，而是将该桶升级为一棵手写红黑树。这使得桶内查找从 O(N) 优化为 O(log K)（K 为桶内元素数量）。 |
| **第三级** | **链表 (LinkedList)**     | **O(M)**     | **精确存储**：在红黑树的节点中，如果出现 **“完全相同的 Key”** （例如重复插入或极端哈希碰撞导致比较结果为 0），则在该红黑树节点下挂载链表，用于存储不同的 Value 版本或解决最终极的键值冲突。 |

------

##  技术实现亮点

### 1. 纯手写数据结构（零 STL 依赖）

- **哈希表**：采用**链地址法**，动态扩容，负载因子控制在 0.75 以下。
- **红黑树**：完全手写插入、删除、左旋、右旋及颜色翻转逻辑。**拒绝使用 `std::map`**，以证明对底层实现的绝对掌控。

### 2. 性能稳定性保障

- **最坏情况保证**：即使所有文件哈希到同一个桶，整体查询复杂度依然是 **O(log N)**（红黑树），而非 O(N)。
- **双向转换**：当红黑树节点数少于阈值（如 < 8）时，维持红黑树结构；当节点过少时保持红黑树（不退化回链表），避免频繁的结构转换开销。

## 未来优化方向（Roadmap）

1. **无锁并发**：引入读写锁（`std::shared_mutex`）分片，支持多线程高并发元数据读写。
2. **持久化 WAL**：将元数据变更写入预写日志（WAL），服务重启后可快速重建内存索引。
3. **自适应结构转换**：当红黑树节点数超过阈值时，自动转换为跳表（SkipList）以进一步提升并发性能。
